DocTalk Appointments Summary

Backend rules
- Appointments are handled in [backend/api/appointments.py](backend/api/appointments.py) and [backend/services/appointment_service.py](backend/services/appointment_service.py).
- Only doctors can create availability slots.
- Only patients can book appointments.
- Two booking modes exist:
  - Direct booking: patient selects an open slot and must provide `slotId` plus a reason.
  - Open booking: patient chooses a doctor and submits a request with a reason; the appointment starts as `PENDING`.
- Slot validation is strict:
  - direct booking fails if the slot does not exist, is inactive, or is already booked.
  - a second booking for the same slot is blocked by both the slot record and an appointment lookup.
- Doctor actions are rule-based:
  - only the owning doctor can accept or reject a request.
  - accepting a request requires `assignedDate` and converts the appointment to `CONFIRMED`.
  - rejecting a request converts the appointment to `REJECTED`.
- Cancellation is ownership-based:
  - patients can cancel their own appointments.
  - doctors can cancel their own appointments.
  - completed, cancelled, and rejected appointments cannot be cancelled again.
  - if the appointment used a slot, the slot is reset and marked inactive/unbooked.
- Appointment status normalization supports these states: `PENDING`, `CONFIRMED`, `REJECTED`, `COMPLETED`, `CANCELLED`.
- The backend also exposes compatibility routes for older clients, including `/api/my_appointments`, `/api/doctor_dashboard_data`, and `/api/appointments/doctor_schedule_request`.

Database info
- The appointment flow uses Prisma models in [prisma/schema.prisma](prisma/schema.prisma).
- Main tables and mappings:
  - `patients` -> `Patient` with primary key `username`.
  - `doctors` -> `Doctor` with primary key `doctorId`.
  - `doctor_slots` -> `DoctorSlot` with primary key `id`.
  - `appointments` -> `Appointment` with primary key `id`.
- Key appointment columns:
  - `patientUsername`, `doctorId`, `slotId`, `appointmentDate`, `scheduledTime`, `date`, `time`, `reason`, `note`, `doctorMessage`, `status`, `requestedAt`, `createdAt`, `updatedAt`, `completedAt`.
- Key slot columns:
  - `doctorId`, `startTime`, `endTime`, `isBooked`, `isActive`.
- Relations:
  - each appointment belongs to one patient and one doctor.
  - an appointment may link to one doctor slot through `slotId`.
  - a slot belongs to one doctor and may have one appointment.
  - patients and doctors both keep appointment arrays for reverse lookup.
- Database constraints that matter for appointments:
  - `Appointment.slotId` is unique, so one slot cannot back multiple appointments.
  - `DoctorSlot` is indexed by `doctorId` and by `doctorId + startTime` for schedule queries.
  - `Appointment` is indexed by `patientUsername`, `doctorId`, `slotId`, and `status` for history and dashboard filtering.
- Stored defaults are important too:
  - slots default to `isBooked = false` and `isActive = true`.
  - appointments default to `status = PENDING` and have `requestedAt` / `createdAt` timestamps.

Frontend behavior
- Patient appointment UI lives in [frontend/src/pages/PatientDashboard.jsx](frontend/src/pages/PatientDashboard.jsx).
- The patient view loads appointments with `/api/my_appointments` first, then falls back to `/api/appointments`.
- The patient booking form supports both direct and open booking.
- A doctor is auto-selected when available, and available slots are fetched whenever the doctor changes.
- The UI prevents submission unless a doctor and reason are present.
- Direct bookings use a selected slot; open requests do not require a slot.
- If booking returns a `409`, the UI refreshes the slot list and tells the patient that the slot was already taken.
- Patients can cancel their appointments from the appointment list.
- The patient doctor-chat flow can create a consultation from a booked appointment when needed.
- Doctor appointment UI lives in [frontend/src/pages/DoctorDashboard.jsx](frontend/src/pages/DoctorDashboard.jsx).
- The doctor dashboard separates the workflow into requests, upcoming sessions, completed sessions, and slot management.
- Requests require the doctor to choose a date/time before accepting.
- The doctor can accept or decline requests, and the dashboard refreshes after each action.
- Slot management only allows open, non-past, non-booked time blocks to be toggled and saved.
- Upcoming sessions can be marked complete or cancelled from the dashboard.

Current rule-based system behavior
- The system is not a free-form appointment editor.
- Booking choice, slot availability, ownership, and status transitions are all constrained by backend checks.
- The frontend mirrors those checks with disabled controls, fallback loading, and explicit confirmations for cancellation and scheduling.
